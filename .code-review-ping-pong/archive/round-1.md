#### Issue 1.1 — Learning extractor writes to two different output directories
- **File:** `skills/learning-extractor/SKILL.md`
- **Line:** 26-34
- **Code:**
  ```md
  Analisa o que aconteceu nesta sessao, identifica pontos de melhoria em qualquer artefato envolvido (rules, memory, skills, agents, docs, workflows), e gera um relatorio acionavel salvo em `squads/kaizen-v2/data/intelligence/learnings/` para revisao e aplicacao em batch.

  - Com slug: `squads/kaizen-v2/data/intelligence/learnings/2026-04-03-db-delivery-learning.md`
  ```
- **Problem:** The skill points to `squads/kaizen-v2/data/intelligence/learnings/`, while the task, Forge plugin, Claude command, and current directory layout point to `squads/kaizen-v2/data/learnings/`. This splits the contract in two places: reports generated by one entrypoint will not be where the other entrypoints expect them, and operators cannot know which location is authoritative.
- **Suggestion:**
  ```md
  Use one canonical path everywhere, for example:
  `squads/kaizen-v2/data/learnings/YYYY-MM-DD-{slug}-learning.md`
  ```

#### Issue 1.2 — Claude slash commands depend on a hardcoded repo path
- **File:** `.claude/commands/learning-extractor.md`
- **Line:** 1
- **Code:**
  ```md
  Read the skill at `~/aios-core/skills/learning-extractor/SKILL.md` and follow its instructions exactly.
  ```
- **Problem:** Both `.claude/commands/learning-extractor.md` and `.claude/commands/critica.md` assume the repository lives at `~/aios-core`. That works only on this machine layout. #### Issue 1.3 — Kaizen task promises automatic apply, but the skill requires explicit user approval
- **File:** `squads/kaizen-v2/tasks/extract-session-learnings.md`
- **Line:** 23-30
- **Code:**
  ```md
  - Actionable improvements applied to system artifacts (rules, memory, skills, etc.)
  - Session learning report saved to `data/learnings/YYYY-MM-DD-session.md`

  - [ ] Improvements applied to target artifacts (not just documented)
  ```
- **Problem:** The task contract says `*learn` applies improvements as part of the task, and the README/plugin mirror the same expectation. The skill does not do that by default: it generates a report first, waits for user review, and only runs Phase 5 after the user explicitly responds `aplicar`. That mismatch breaks post-reflect automation and creates a false expectation that invoking `*learn` is safe to treat as a one-step mutating workflow.
- **Suggestion:**
  ```md
  Either document `*learn` as "generate report, then wait for explicit aplicar"
  or update the skill/integration to support a true automated apply mode.
  ```

#### Issue 1.4 — Memory Keeper resolves requests to an undefined *briefing command
- **File:** `squads/kaizen-v2/agents/memory-keeper.md`
- **Line:** 41-47
- **Code:**
  ```yaml
  - "show briefing" / "what should I know" → *briefing → read last briefing from patterns.yaml
  ```
- **Problem:** `REQUEST-RESOLUTION` routes users to `*briefing`, but `command_loader` and `commands` only define `*capture`, `*patterns`, `*reflect`, `*learn`, `*archive`, `*status`, `*help`, and `*exit`. A natural-language request can therefore resolve to a command that the same agent file does not implement.
- **Suggestion:**
  ```yaml
  - "show briefing" / "what should I know" → *status → check pattern health and latest briefing context
  ```
