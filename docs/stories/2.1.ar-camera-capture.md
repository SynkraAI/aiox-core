## Story
**As a** user,
**I want** to capture front and back photos of my body using the app camera with alignment guides,
**so that** the AI can analyze my physique accurately from both angles.

## Acceptance Criteria
1. CameraScreen apresenta guia em 2 etapas: foto frontal e foto de costas — com silhueta-guia sobreposta em cada etapa
2. Botão de captura com feedback visual (flash/animação) — foto salva localmente antes do upload
3. Tela de preview por etapa: botão "Refazer" (recaptura só aquela foto) e botão "Confirmar" (avança para próxima)
4. Endpoint `POST /analyses` retorna `{ analysis_id, upload_urls: { front, back } }` com presigned URLs S3 válidas por 5 min
5. App faz upload direto para S3 via PUT para cada presigned URL (frente → costas → confirmação)
6. Após ambos os uploads confirmados: `POST /analyses/{id}/process` dispara o pipeline no ai-engine
7. App navega para loading screen em `/(app)/analysis/[id]` (polling implementado na Story 2.2)
8. Validação de tamanho (máximo 10 MB) e formato (JPEG/PNG) por foto no frontend

## Dev Notes

### Endpoints (API Gateway)
```
POST /analyses           → cria análise + retorna presigned URLs → 201 { analysis_id, upload_urls: { front, back } }
POST /analyses/:id/process → dispara pipeline ai-engine (async)  → 202 { status: 'processing' }
GET  /analyses/:id       → status e resultado                    → 200 { id, status, scores?, report?, workout_plan? }
```

### Schema do banco
```sql
-- analyses
CREATE TABLE analyses (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  status TEXT NOT NULL DEFAULT 'processing'
    CHECK (status IN ('processing', 'completed', 'failed')),
  photo_front_url TEXT,     -- nullificada após deleção pelo ai-engine
  photo_back_url TEXT,      -- nullificada após deleção pelo ai-engine
  photos_deleted_at TIMESTAMPTZ,
  scores JSONB,             -- preenchido pelo ai-engine ao completar
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  completed_at TIMESTAMPTZ
);
CREATE INDEX idx_analyses_user_id ON analyses(user_id);
CREATE INDEX idx_analyses_created_at ON analyses(created_at DESC);
```

### S3 Presigned URLs (API Gateway)
```typescript
import { S3Client, PutObjectCommand } from '@aws-sdk/client-s3'
import { getSignedUrl } from '@aws-sdk/s3-request-presigner'

const s3 = new S3Client({ region: process.env.AWS_REGION })

async function generatePresignedUrl(userId: string, analysisId: string, angle: 'front' | 'back') {
  const key = `uploads/${userId}/${analysisId}/${angle}.jpg`
  const command = new PutObjectCommand({
    Bucket: process.env.S3_BUCKET,
    Key: key,
    ContentType: 'image/jpeg',
  })
  const url = await getSignedUrl(s3, command, { expiresIn: 300 })
  return { url, key }
}
```

#### Camera hardcoded para câmera frontal (HIGH — funcional)

`camera.tsx` linha 133 usa `facing={'front' as CameraType}` para **ambas** as etapas (frente e costas). Para a foto de costas, o usuário precisaria estar de costas para uma câmera frontal — o que não funciona para análise corporal real.

```tsx
// Atual — ambas as fotos na câmera frontal
<CameraView ref={cameraRef} style={styles.camera} facing={'front' as CameraType}>

// Correto
const cameraFacing: CameraType = step === 'front' ? 'front' : 'back'
<CameraView ref={cameraRef} style={styles.camera} facing={cameraFacing}>
```

Impacta diretamente a qualidade das fotos de costas e a viabilidade da análise de IA.

---

#### Validação de tamanho não implementada (MEDIUM — AC8 parcialmente violado)

A constante `MAX_FILE_SIZE_BYTES` está definida em `camera.tsx` mas **nunca usada**. O comentário no código reconhece a limitação mas não há verificação real de tamanho.

AC8 exige: "Validação de tamanho (máximo 10 MB) e formato (JPEG/PNG) por foto no frontend."

Sugestão para resolver: em `handleCapture`, após `takePictureAsync`, obter o tamanho via `expo-file-system`:
```typescript
import * as FileSystem from 'expo-file-system'
const info = await FileSystem.getInfoAsync(photo.uri, { size: true })
if ((info as FileSystem.FileInfo & { size: number }).size > MAX_FILE_SIZE_BYTES) {
  Alert.alert('Foto muito grande', 'Máximo 10 MB por foto.')
  return
}
```

---

#### Presigned URL armazenada no banco (MEDIUM — segurança)

`analyses.ts` armazena a presigned URL completa (incluindo assinatura AWS) em `photo_front_url`/`photo_back_url`. Ao completar a análise, é corretamente nullificada — mas durante o processamento essa URL com credenciais AWS temporárias fica exposta no banco.

Recomendação: armazenar apenas a S3 key (`uploads/{userId}/{analysisId}/front.jpg`) e reconstruir URLs quando necessário.

---

#### Qualidade dos testes (LOW)

Os testes em `analyses.routes.test.ts` não testam as rotas via HTTP (sem `app.inject`). Assertam sobre mocks e dados locais diretamente:

```typescript
// Não testa o handler HTTP — apenas asserta sobre o mock
expect(result.analysis_id).toBe('analysis-uuid-123')
```

O teste de `POST /internal/analyses/:id/complete` está essencialmente vazio (`expect(mockPool.query).toBeDefined()`). Recomenda-se usar `app.inject()` do Fastify para testes de integração reais.

---

#### Race condition no freemium (LOW)

`checkFreemiumLimit` e `INSERT INTO analyses` não estão em uma transação. Dois requests simultâneos de um usuário free poderiam ambos passar o check antes de qualquer um criar a análise.

Recomendação: envolver as operações em `BEGIN/COMMIT` ou usar `SELECT FOR UPDATE` na tabela `users`.
